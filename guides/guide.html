sb-flow Workflow Guide

AI에게 맡기는
체계적인 개발

복잡한 코드를 몰라도 괜찮습니다.
Claude에게 명령어를 복사해서 붙여넣기만 하면 됩니다.

전체 흐름 한눈에 보기

설계SpecKit (1~7단계)
→
연결브릿지 (7.5단계)
→
구현bkit (8~10단계)
→
검증bkit (11~17단계)
원리는 간단합니다. 먼저 "뭘 만들지" 정리하고(설계), 그걸 AI가 읽을 수 있게 정리해주고(연결), AI가 코드를 짜고(구현), 제대로 됐는지 확인합니다(검증). 모든 단계에서 여러분이 할 일은 명령어를 복사해서 Claude에 붙여넣기하는 것뿐입니다.

시작하기 전에

Claude Code 터미널에서 명령어를 입력하는 방법만 알면 됩니다.

📋

복사 & 붙여넣기

각 단계의 명령어 블록을 복사 버튼으로 복사한 뒤, Claude Code 터미널에 붙여넣고 Enter를 누르세요.

💬

Claude와 대화하기

명령어를 넣으면 Claude가 알아서 작업합니다. 중간에 질문이 오면 자연어(한국어)로 답하면 됩니다.

🔵 vs ⚪

필수 vs 선택

필수 표시는 꼭 해야 하는 단계, 선택 표시는 건너뛰어도 됩니다.

📍

현재 단계 확인

어디까지 했는지 모르겠으면 언제든 Claude에 /sb-guide를 입력하세요.

A

설계 구간 — "뭘 만들지 정리하기"

아직 코드를 짜지 않습니다. AI에게 "이런 걸 만들어줘"라고 정확히 알려주는 단계입니다.

1
프로젝트 규칙 정하기 필수
프로젝트 전체에서 지켜야 할 원칙 선언 (최초 1회)
▼
AI가 코드를 만들 때 절대 어기면 안 되는 규칙을 정합니다. 예를 들어 "TypeScript만 써라", "한글로 문서를 써라" 같은 것들이에요. 프로젝트당 처음 한 번만 하면 됩니다.
Claude에 붙여넣기
/speckit.constitution 우리 프로젝트의 핵심 원칙을 제정해줘: - TypeScript Strict Mode + Next.js App Router - 모든 문서는 한글로 작성 - (여기에 원하는 규칙 추가)
💡
규칙은 자유롭게 적으세요. "코드에 주석 많이 달아줘", "디자인은 심플하게" 같은 것도 됩니다. 나중에 수정할 수도 있어요.
2
만들고 싶은 것 설명하기 필수
어떤 기능을 만들고 싶은지 자연어로 설명
▼
친구한테 설명하듯이 만들고 싶은 기능을 편하게 한국어로 적으면 됩니다. AI가 체계적인 명세서로 정리해줍니다.
Claude에 붙여넣기
/speckit.specify (여기에 만들고 싶은 것을 자유롭게 설명) 예시: 포모도로 타이머를 만들고 싶어. 25분 집중하고 5분 쉬는 사이클이야. 타이머 시작/정지 버튼이 있고, 오늘 몇 번 집중했는지 보여줬으면 좋겠어.
⚠️
Claude가 [NEEDS CLARIFICATION] 표시를 남기면 → 3단계로 가세요.
표시가 없으면 → 3단계 건너뛰고 4단계로 가세요.
3
애매한 부분 해소하기 선택
Claude의 질문에 답변하면 명세가 더 정확해집니다
▼
2단계에서 설명이 부족했던 부분을 Claude가 물어봅니다. 한국어로 자연스럽게 답하면 됩니다.
Claude에 붙여넣기
/speckit.clarify
💡
Claude가 질문하면 평소 말하듯이 대답하세요. 예: "로그인 없이 브라우저에만 저장할게"
4
기술 설계 만들기 필수
AI가 데이터 구조, API, 기술 스택을 자동 설계
▼
2단계에서 "뭘 만들지" 정했다면, 이제 "어떤 기술로 만들지"를 AI가 결정합니다. 데이터베이스 구조, API 설계 등을 자동으로 만들어줍니다.
Claude에 붙여넣기
/speckit.plan
5
할 일 목록 만들기 필수
설계를 구체적인 작업 목록(체크리스트)으로 분해
▼
설계 문서를 읽고 "T001: ○○ 파일 만들기", "T002: △△ 기능 구현" 같은 체크리스트를 만들어줍니다. 나중에 코드 짤 때 이 순서대로 진행합니다.
Claude에 붙여넣기
/speckit.tasks
6
품질 체크리스트 선택
요구사항이 잘 작성됐는지 추가 검증
▼
명세서의 품질을 검증하는 체크리스트를 만듭니다. 건너뛰어도 크게 문제없습니다.
Claude에 붙여넣기
/speckit.checklist
7
설계 검증하기 선택
명세 ↔ 설계 ↔ 할 일 목록 간 모순이 없는지 확인
▼
지금까지 만든 문서들이 서로 모순되지 않는지 AI가 교차 검증합니다. 문서를 수정하지 않고, 문제점만 알려줍니다.
Claude에 붙여넣기
/speckit.analyze
B

브릿지 — 설계를 AI에게 전달하기

설계 문서를 하나의 파일로 요약해서 코드를 짜는 AI에게 넘깁니다.

7.5
브릿지 문서 만들기 필수
설계 산출물을 SPEC_CONTEXT.md 하나로 요약
▼
지금까지 만든 설계 파일들을 하나의 요약 파일(SPEC_CONTEXT.md)로 합칩니다. 이후 모든 단계에서 이 파일 하나만 참조하면 됩니다.
Claude에 붙여넣기 — {기능명}을 실제 이름으로 바꾸세요
specs/ 디렉토리의 {기능명} 폴더에 있는 constitution.md, spec.md, plan.md, data-model.md, contracts/api.md, tasks.md를 모두 읽고 SPEC_CONTEXT.md 템플릿의 각 섹션을 채워줘.
⚠️
이 단계 이후부터 SpecKit 명령어는 쓰지 않습니다. 구현과 검증은 전부 bkit(아래 단계들)이 담당합니다.
C

구현 구간 — "실제로 만들기"

설계 문서를 바탕으로 AI가 실제 코드를 작성합니다.

8
개발 계획 세우기 필수
어떤 순서와 전략으로 코드를 짤지 계획
▼
설계가 "무엇을 만들지"였다면, 이건 "어떤 순서로 만들지" 계획하는 단계입니다.
Claude에 붙여넣기 — {기능명}을 실제 이름으로 바꾸세요
/pdca plan {기능명} SPEC_CONTEXT.md 파일을 참조해서 개발 계획을 세워줘. @vercel-react-best-practices 가이드라인을 읽고 이 기준에 맞춰 개발 계획을 세워줘. 다음 세 가지 기준을 반드시 지켜야 해: 1. 헌법(constitution)의 모든 원칙을 준수할 것 2. spec.md의 모든 기능 요구사항(FR)을 빠짐없이 구현할 것 3. tasks.md의 모든 태스크를 완료 기준으로 삼을 것
9
상세 설계 만들기 필수
컴포넌트, 함수 구조까지 구체적으로 설계
▼
Claude에 붙여넣기
/pdca design {기능명} SPEC_CONTEXT.md 파일을 참조해서 작업해줘. @vercel-react-best-practices 가이드라인을 읽고 이 기준에 맞춰 상세 설계를 작성해줘.
10
코드 작성하기 필수
AI가 실제 코드를 작성합니다
▼
드디어 코드가 만들어집니다! AI가 설계 문서를 보고 파일을 하나씩 생성합니다. 시간이 좀 걸릴 수 있어요.
Claude에 붙여넣기
/pdca do {기능명} SPEC_CONTEXT.md 파일을 참조해서 작업해줘. @vercel-react-best-practices 가이드라인을 읽고 이 기준에 맞춰 구현해줘.
🎉
여기까지 완료되면 실제로 동작하는 코드가 생성됩니다!
▶️
서버 시작: start.bat을 실행해서 브라우저에서 결과를 확인하세요. (macOS/Linux: npm run dev)
10.5
대화하며 수정하기 필수
마음에 안 드는 부분을 Claude와 대화하며 고치기
▼
코드가 만들어졌지만 마음에 안 드는 부분이 있을 거예요. 자유롭게 한국어로 수정을 요청하세요. 수정이 끝나면 문서도 업데이트합니다.
자유롭게 요청하세요 (예시)
배경색을 더 어둡게 바꿔줘 버튼 크기를 좀 키워줘 이 기능 추가해줘: ...
수정이 다 끝나면
문서 동기화 — Claude에 붙여넣기
지금까지 대화에서 코드에 적용된 변경 사항을 모두 파악해서 bkit plan/design 문서를 현재 코드 상태에 맞게 업데이트해줘. 변경된 항목 목록도 알려줘.
D

검증 구간 — "제대로 됐는지 확인하기"

코드가 설계대로 만들어졌는지 확인하고, 부족한 부분을 자동으로 고칩니다.

11
코드 리뷰 선택
코드 품질과 보안을 검토
▼
Claude에 붙여넣기
/code-review
12
설계 vs 코드 비교하기 필수
설계대로 만들어졌는지 자동 검사 (Match Rate 계산)
▼
AI가 설계 문서와 실제 코드를 비교해서 몇 퍼센트 일치하는지(Match Rate) 계산합니다. 90% 이상이면 합격!
Claude에 붙여넣기
/pdca analyze {기능명}
📊
결과가 90% 이상이면 → 14단계로 건너뛰세요
결과가 90% 미만이면 → 바로 다음 13단계를 실행하세요
13
자동 개선하기 조건부
Match Rate가 90% 미만일 때만 실행
▼
AI가 부족한 부분을 자동으로 고치고 다시 검사합니다. 최대 5번 반복하며, 90%에 도달하면 자동으로 멈춥니다.
Claude에 붙여넣기
/pdca iterate {기능명}
14
QA 검증 선택
실제로 브라우저에서 테스트하기 위한 가이드
▼
테스트 시나리오를 만들어줍니다. 실제 테스트는 여러분이 브라우저에서 직접 해야 합니다.
Claude에 붙여넣기
/zero-script-qa
15
코드 정리 선택
중복 제거, 불필요한 코드 정리
▼
Claude에 붙여넣기
/simplify
16
완료 보고서 필수
전체 개발 사이클 요약 보고서 생성
▼
Claude에 붙여넣기
/pdca report {기능명}
16.5
세션 아카이브 선택
완료된 PDCA 문서를 아카이브 폴더로 정리
▼
Claude에 붙여넣기
/pdca archive {기능명}
E

동기화 — 문서를 현실에 맞추기

코드가 완성됐으니, 처음 만든 설계 문서를 최종 상태로 업데이트합니다.

17
문서 사후 동기화 필수
설계 문서를 실제 코드에 맞게 업데이트
▼
개발하면서 원래 계획에 없던 기능이 추가되거나, 기존 기능이 바뀌었을 수 있습니다. 설계 문서를 현재 코드에 맞게 업데이트합니다. 다음에 기능을 추가할 때 정확한 출발점이 됩니다.
Claude에 붙여넣기 — SpecKit 사용 시
현재 구현된 코드와 원래 specs/NNN-기능명/을 비교해서 아래 네 가지를 모두 처리하고 구현 코드와 비교 후 spec.md, plan.md, tasks.md를 업데이트: 1. 완료된 기존 태스크: [ ] → [x] 2. 수정된 기존 기능: 해당 FR을 실제 구현에 맞게 업데이트 3. 새로 추가된 기능: spec.md에 새 FR 추가 + tasks.md에 새 태스크 추가 후 [x] 4. 제거된 기능이 있다면: 해당 FR에 deprecated 표시 처리 완료 후 변경된 항목 목록을 케이스별로 요약해줘.
그 다음
설계 문서 간 검증
/speckit.analyze
마지막으로
브릿지 문서 최종 갱신
방금 업데이트한 spec.md와 tasks.md 내용을 반영해서 SPEC_CONTEXT.md의 기능 명세 요약, 도메인 규칙, 태스크 현황 섹션을 최종 완료 상태로 갱신해줘.
🎉
축하합니다! 전체 워크플로우가 완료됐습니다. 다음 기능을 추가하려면 2단계부터 다시 시작하면 됩니다.

자주 묻는 질문

모든 단계를 다 해야 하나요? ▼
아닙니다. 필수 표시된 단계만 하면 됩니다. 최소 경로: 1→2→4→5→7.5→8→9→10→12→16→17 (11단계)
{기능명}에는 뭘 적어야 하나요? ▼
만들고 있는 기능의 이름을 짧게 적으세요. 예: 포모도로-타이머, 회원가입, 월간통계. 한국어도 됩니다.
중간에 실수했으면 어떻게 하나요? ▼
걱정 마세요. 같은 명령어를 다시 실행하면 됩니다. Claude가 기존 파일을 확인하고 수정 여부를 물어봅니다.
작은 버그 수정도 이 전체 과정을 거쳐야 하나요? ▼
아닙니다. 단순 버그 수정이나 UI 수정은 Claude에게 바로 말하면 됩니다. 이 워크플로우는 새 기능 개발에 사용하세요.
@vercel-react-best-practices는 뭔가요? ▼
Vercel이 만든 React 코드 품질 가이드라인입니다. 프로젝트 시작 시 한 번 설치하면 자동으로 참조됩니다. 설치 방법은 프로젝트의 SPECKIT_BKIT_WORKFLOW.md에 있습니다.
