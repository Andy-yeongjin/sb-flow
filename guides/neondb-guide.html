Connection String 두 개 복사 사람

Prisma는 Pooled(앱 런타임용)와 Direct(마이그레이션용) 두 개가 모두 필요합니다.

🔒
이 문자열에는 비밀번호가 포함되어 있습니다. Slack/Discord/GitHub 등 외부에 절대 붙여넣지 마세요. 노출 시 즉시 NeonDB → Settings → Reset password.

프로젝트 생성 직후 화면에서 Connection String 클릭

① Connection pooling 토글 ON 상태에서 하단 Copy snippet 클릭 → POOLED URL 로 임시 보관

Pooled URL 예시 (앱 런타임이 사용)
postgresql://username:password@ep-xxxxx-pooler.ap-southeast-1.aws.neon.tech/neondb?sslmode=require

② Connection pooling 토글 OFF 후 하단 Copy snippet 다시 클릭 → DIRECT URL 로 임시 보관

Direct URL 예시 (마이그레이션이 사용)
postgresql://username:password@ep-xxxxx.ap-southeast-1.aws.neon.tech/neondb?sslmode=require
📝
두 URL의 차이는 호스트에 -pooler가 붙는지뿐입니다. 메모장에 임시로 붙여두고 다음 단계로 가세요.
4

Claude에게 연결 시키기 프롬프트 복사

아래 프롬프트를 Claude Code 대화창에 붙여넣고, {{POOLED_URL}}·{{DIRECT_URL}} 두 군데만 3단계에서 복사한 값으로 교체하세요.

Claude Code 대화창에 붙여넣기 → URL 두 개만 교체
constitution.md를 읽고 모든 원칙을 준수해줘. 특히 보안: API 키/DB 자격증명은 코드에 하드코딩 금지, .env 로만 관리, .gitignore에 .env 포함 확인. NeonDB(PostgreSQL)를 이 프로젝트에 Prisma ORM으로 연결해줘. 현재 이 프로젝트는 sqlite로 개발 중일 가능성이 높으니, 기존 sqlite 스키마와 모델 정의를 반드시 먼저 확인하고 그대로 이관해야 한다. 아래 절차를 순서대로 수행해: 0. 현재 상태 점검 (가장 먼저) - prisma/schema.prisma 가 있는지 확인하고 있으면 datasource provider, model 정의를 모두 읽어줘 - 코드 내에서 prisma.* 호출이 어떤 모델·필드를 사용하는지 grep으로 파악 - 위 두 가지를 본 뒤 이관 계획을 한 줄로 요약해서 알려줘 (어떤 모델 N개를 어떻게 옮길지) 1. 의존성 확인 - prisma (devDependency), @prisma/client 설치 여부 확인 (없으면 설치) - prisma/schema.prisma 가 이미 있으면 그걸 기준으로 작업, 없으면 npx prisma init 2. .env 파일에 아래 두 환경변수 작성 (이미 있으면 갱신): DATABASE_URL="{{POOLED_URL}}" DIRECT_URL="{{DIRECT_URL}}" 기존에 sqlite용 DATABASE_URL="file:./dev.db" 가 있었다면 위 값으로 교체. 3. prisma/schema.prisma 수정: - datasource db 블록의 provider 를 "postgresql"로 (현재 "sqlite"였다면 교체) - url을 env("DATABASE_URL")로 설정 - 같은 블록에 directUrl = env("DIRECT_URL") 추가 - 기존 model 정의는 가능한 그대로 유지. sqlite ↔ postgresql 간 타입 차이가 있는 부분만 PostgreSQL에 맞게 미세 조정 (Int autoincrement 동작, Bytes, DateTime 정밀도, @default(now()) 등) - 정의된 model이 하나도 없을 때만, 메모리/파일/JSON으로 관리되던 데이터 구조를 분석해서 새로 모델 정의 4. 기존 sqlite 마이그레이션 정리: - prisma/migrations/ 폴더가 sqlite 기준으로 만들어져 있다면 PostgreSQL과 호환되지 않으므로 폴더 통째로 삭제 - 로컬 dev.db, *.db, *.db-journal, prisma/dev.db 가 .gitignore에 포함되어 있는지 확인 (없으면 추가) 5. 마이그레이션 실행: - npx prisma migrate dev --name init - npx prisma generate 6. lib/prisma.ts 싱글톤 클라이언트 생성 — Next.js dev 모드 hot reload 시 연결 풀 누수 방지 패턴 적용 (globalThis 캐싱). 이미 있으면 건너뜀. 7. 기존에 메모리/파일/sqlite로 데이터를 읽고 쓰던 부분을 모두 Prisma 클라이언트 호출로 교체. Server Actions / Route Handlers 안에서만 사용하고, 클라이언트 컴포넌트에는 절대 import하지 말 것. 8. .gitignore에 .env, .env*.local 포함 확인 (없으면 추가). 9. npm run build로 프로덕션 빌드 검증 (constitution 제10조). 작업 후 어떤 파일이 추가/변경됐고, sqlite에서 가져온 모델이 무엇인지, 새로 추가한 모델이 무엇인지 요약해줘.
⚠️
붙여넣기 전 체크: URL의 username:password 부분이 그대로 들어가 있어야 합니다. NeonDB 콘솔에서 Hide password 토글이 켜진 상태로 복사하면 비밀번호가 빠진 채 복사돼요.

연결 확인 프롬프트 복사

Claude에게 프롬프트로 검증을 시키고, 결과 보고를 받으세요.

Claude Code 대화창에 붙여넣기 — 자동 검증
방금 연결한 NeonDB가 정상 동작하는지 다음 절차로 검증하고 결과를 표로 보고해줘: 1. 연결 ping - lib/prisma.ts (또는 동일한 싱글톤)을 import한 임시 스크립트 scripts/check-db.ts 를 만들어서 await prisma.$queryRaw`SELECT 1 as ok` 실행 → 결과 출력 - tsx 또는 ts-node로 실행: npx tsx scripts/check-db.ts 2. 마이그레이션 상태 점검 - npx prisma migrate status 실행해서 "Database schema is up to date" 가 나오는지 확인 3. 모델별 row count - schema.prisma 안의 모든 model에 대해 prisma..count() 를 호출해서 | 모델명 | row count | 비고 | 형식의 표로 정리 4. 임시 INSERT/SELECT/DELETE 라운드트립 - 가장 단순한 모델 하나(예: User 같은 단일 PK 모델)를 골라 테스트용 row 1개 insert → 같은 조건으로 select 해서 일치 확인 → delete - 모델이 없으면 _HealthCheck 라는 임시 모델을 만들지는 말고 그냥 3번까지만 하고 4번은 "스킵: 적절한 모델 없음"으로 보고 5. 종료 정리 - scripts/check-db.ts 는 작업이 끝나면 삭제 (또는 원하면 유지하고 .gitignore에 추가 여부 물어봐줘) - 어떤 단계에서 어떤 출력이 나왔는지 그대로 보고 - 실패가 있다면 에러 코드(P1001/P1003/P2021 등)와 가장 가능성 높은 원인 한 줄 마지막에 "✅ 연결 정상 / ⚠️ 부분 정상 / ❌ 실패" 중 하나로 한 줄 결론을 내려줘.
📊
Claude가 위 프롬프트로 검증 결과 표를 출력합니다. 모든 model이 row count로 잡히고 라운드트립까지 통과하면 연결·스키마·권한 모두 정상이라는 뜻이에요.
👀
UI로도 직접 보고 싶다면 NeonDB 대시보드 → 좌측 Tables 탭에서 동일한 테이블이 생성됐는지 확인할 수 있어요 (선택).

자주 묻는 질문

Pooled / Direct 두 개를 왜 다 받나요? ▼
Pooled (DATABASE_URL): 앱 런타임이 사용. Vercel 서버리스 함수가 매번 새 연결을 만들어 NeonDB를 터뜨리지 않도록 풀링된 엔드포인트로 보냅니다.

Direct (DIRECT_URL): prisma migrate가 사용. 마이그레이션은 풀링된 커넥션 위에서 실행할 수 없어서 직접 연결이 별도로 필요합니다.
무료 플랜 한계는? ▼
0.5 GB 스토리지, 1개 프로젝트, 미사용 시 자동 슬립. 초기 개발/소규모 프로덕션에 충분합니다. 슬립 상태에서 첫 요청은 약간 느릴 수 있습니다 (cold start).
프로덕션 배포 시 환경변수는 어디에? ▼
Vercel 배포 가이드 4단계에서 동일한 DATABASE_URL / DIRECT_URL 두 변수를 Vercel 프로젝트 환경변수에 등록합니다.
Drizzle을 쓰고 싶어요 ▼
3단계에서 받는 connection string은 ORM과 무관한 표준 PostgreSQL URL이라 그대로 쓰면 됩니다. 4단계 프롬프트에서 "Prisma ORM" → "Drizzle ORM"으로 바꾸고 절차도 Drizzle 기준(drizzle-kit, drizzle.config.ts, drizzle/migrations/)으로 다시 적어 시켜주세요.
자격증명을 실수로 GitHub에 푸시했어요 ▼
즉시 NeonDB 대시보드 → Settings → Reset password로 비밀번호 재발급 후 새 connection string으로 .env 갱신하세요. GitHub에서 파일을 지우는 것만으로는 git history에 남아 누출된 상태입니다. git filter-repo로 history를 다시 쓰거나, 새 저장소로 옮기세요.
"P1001: Can't reach database" 에러가 나요 ▼
① URL 끝에 ?sslmode=require가 붙어 있는지 확인 (NeonDB 필수)
② 무료 플랜 슬립 상태일 수 있음 — 잠시 후 재시도
③ 회사 방화벽이 5432 포트를 막고 있을 수 있음 — 핫스팟 등으로 테스트
