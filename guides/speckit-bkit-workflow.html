SpecKit + bkit
통합 워크플로우 가이드

Spec-Driven Development(SDD) 기반 개발 프로세스

설계부터 구현, 검증, 보고까지 — 완전한 개발 사이클

SpecKit과 bkit PDCA를 결합한 통합 워크플로우를 소개합니다. SDD 방법론 기반으로 설계와 구현을 완전히 분리하여 품질을 보장합니다.

두 프레임워크의 역할 분리

SpecKit — 설계 전담

1 ~ 7.5단계

  • 헌법 제정 → 명세 → 기술 설계
  • 태스크 분해 → 브릿지 문서

"무엇을, 왜 만드는가?"

bkit PDCA — 구현·검증 전담

8 ~ 16.5단계

  • 개발 계획 → 구현 → 갭 분석
  • 개선 → 브라우저 검증 → 보고 → 문서 동기화

"어떻게 만들고 제대로 됐는가?"

역할 분리 원칙: 두 프레임워크는 SPEC_CONTEXT.md를 경계로 완전히 분리

SpecKit이 설계를 완료하면 bkit이 이어받아 구현·검증·보고를 끝까지 담당

/sb-oneshot 하나로 설계 ~ 브라우저 검증 ~ 문서 동기화까지 14단계 자동 완주

핵심은 역할 분리입니다. SpecKit은 "무엇을 왜 만드는가"에 집중하고, bkit은 "어떻게 만들고 제대로 됐는가"에 집중합니다. SPEC_CONTEXT.md가 두 프레임워크 간의 유일한 연결점입니다.

전체 플로우차트

준비 디자인 시스템 (선택·최초 1회)
.pen 파일 → designs/ → sb-design → design.md + design-tokens.css
A SpecKit 설계 구간 (1~7.5)
constitution → specify → plan → tasks → SPEC_CONTEXT.md
B bkit 구현 구간 (8~10.5)
pdca plan → pdca design → pdca do → 반복 개선
C bkit 검증·보고 구간 (12~16.5)
analyze → iterate → 브라우저 검증 → simplify → report → archive
D 동기화 (17)
문서 사후 동기화 → SPEC_CONTEXT.md 갱신
전체 흐름은 4개 구간으로 나뉩니다. A구간에서 설계를 완료하고 SPEC_CONTEXT.md로 핸드오프, B구간에서 구현, C구간에서 검증과 보고, D구간에서 문서를 역동기화합니다.

bkit은 SpecKit 파일을 자동으로 읽지 않는다

SpecKit 설계 산출물을 bkit에 전달하는 3가지 방법

방법 A (권장)

명령어 실행 시 파일 경로를 직접 나열

방법 B

CLAUDE.md Spec References 섹션에 경로 추가

서브에이전트에는 불충분

방법 C (가장 편리)

SPEC_CONTEXT.md 브릿지 문서로 모든 산출물 요약 전달

상황A 필요?B 필요?C 충분?
/pdca plan, /pdca design없어도 동작권장충분
/pdca analyze, /pdca iterate반드시 필요필요충분
/pdca report없어도 동작권장충분
가장 중요한 주의사항입니다. bkit은 SpecKit 파일을 자동으로 읽지 않으므로, 반드시 전달 방법을 선택해야 합니다. 방법 C(SPEC_CONTEXT.md)가 가장 편리합니다.
A 설계 구간

SpecKit 1~3단계

1 헌법 제정 — /speckit.constitution

  • 프로젝트 전체 불변 원칙 선언 (프로젝트당 최초 1회)
  • 예: "모든 비즈니스 로직은 순수 함수로 격리", "TypeScript Strict Mode 필수"
  • 시맨틱 버저닝으로 관리, 이후 모든 SpecKit 명령이 헌법 참조

2 개발 명세서 — /speckit.specify

  • 기술 독립적 요구사항 정의 — "사용자가 무엇을 할 수 있어야 하는지"
  • User Stories, 기능 요구사항(FR), 성공 기준, 에지 케이스 자동 생성
  • 모호한 부분은 [NEEDS CLARIFICATION] 마커로 표시

3 모호성 해소 — /speckit.clarify (선택)

  • 8개 차원 스캔: 기능 범위, 데이터 모델, UX 흐름, 품질 속성, 통합, 에지 케이스, 제약, 용어
  • 최대 5개 우선순위 질문 → 답변 즉시 반영
설계 구간의 첫 3단계입니다. 헌법은 프로젝트 최초 1회, 명세서는 기능마다 작성합니다. 3단계는 선택이지만, 모호한 명세는 나중에 재작업이 커지므로 권장합니다.
A 설계 구간

SpecKit 4~7.5단계

4 기술 설계 — /speckit.plan

  • Phase 0 리서치: 기술 스택 결정, 라이브러리 비교
  • Phase 1 설계: 데이터 모델, API 계약, 테스트 시나리오
  • 산출물: plan.md, data-model.md, contracts/api.md

5 태스크 분해 — /speckit.tasks

  • Phase별 실행 가능한 태스크 생성
  • Setup → Foundational → User Stories → Polish
  • 모든 태스크에 파일 경로 포함

6 품질 체크리스트 (선택)

요구사항 작성 품질 검증

7 설계 일관성 분석

6가지 탐지: 중복, 모호성, 미명세, 헌법 정합성, 커버리지 갭, 불일치

7.5 브릿지 문서

SPEC_CONTEXT.md 작성
SpecKit → bkit 핸드오프
/sb-bridge 로 자동 생성

0단계 (선택·최초 1회)

/sb-design
.pen → design.md
+ design-tokens.css
design-constitution 자동 검증

7.5단계 이후 SpecKit 명령어는 사용하지 않음 — 구현은 반드시 /pdca do

4단계에서 기술 아키텍처를 결정하고, 5단계에서 실행 가능한 태스크로 분해합니다. 7.5단계의 SPEC_CONTEXT.md가 설계 구간의 마지막이자 구현 구간으로의 핸드오프 지점입니다.
B 구현 구간

bkit PDCA 8~10.5단계

8 PDCA 개발 계획 — /pdca plan

  • SpecKit 설계를 기반으로 bkit PDCA 형식의 개발 계획 작성
  • "어떤 순서와 전략으로 구현할지"에 집중
  • SPEC_CONTEXT.md + @vercel-react-best-practices 참조

9 상세 설계 — /pdca design

  • 컴포넌트 구조 & Props 인터페이스
  • 함수 시그니처, 상태 관리 전략
  • 에러 핸들링 패턴

10 코드 구현 — /pdca do

  • plan/design 문서 기반 구현
  • 헌법·도메인 규칙·API 계약 준수
  • SPEC_CONTEXT.md 함께 전달

10.5 반복 개선 & bkit 문서 동기화

마이너 UI

문서 업데이트 불필요

로직·UX 변경

design.md 수정

신규 기능 추가

plan.md + design.md 추가

구현 구간에서는 SpecKit 명령어를 사용하지 않습니다. 10.5단계가 중요한데, 반복 개선 중 변경된 내용을 bkit 문서에 동기화해야 이후 갭 분석이 정확합니다.
C 검증·보고 구간

12~16.5단계

12 갭 분석 — /pdca analyze

  • gap-detector 에이전트 실행
  • 설계 vs 구현 비교 → Match Rate 산출
  • ≥ 90% → 14단계 / < 90% → 13단계

13 자동 개선 — /pdca iterate

  • pdca-iterator 에이전트
  • Generator-Evaluator 패턴
  • 최대 5회 반복, ≥ 90% 도달 시 중단

14 브라우저 검증 (자동)

Playwright / browser_subagent로 기능·디자인 검증

design-constitution 준수 여부 자동 확인 + 자체 수정

15 코드 정리 — /simplify (선택)

중복 제거, 불필요한 복잡성 제거, 패턴 일관성 확보

14 QA 검증 (선택)

/zero-script-qa — 가이드 도구

Docker 환경에서 qa-monitor 자동 실행 가능

16 완료 보고서 — /pdca report

전체 PDCA 사이클 요약, Match Rate, 4관점 가치 테이블

16.5 세션 아카이브 — /pdca archive

bkit 문서를 docs/archive/YYYY-MM/ 로 이동

검증 구간의 핵심은 Match Rate입니다. 90% 이상이면 통과, 미달이면 자동 개선을 반복합니다. 최대 5회까지 반복 후에도 미달이면 수동 개입이 필요합니다.
D 동기화

17단계 — 문서 사후 동기화

SDD 피드백 루프의 닫힘점 — 다음 기능 개발의 정확한 출발점 보장

/sb-sync

명령어 하나로 모든 문서 자동 동기화

  • bkit plan/design 문서 동기화
  • tasks.md — [ ] → [x], 새 태스크 추가
  • spec.md — FR 수정·추가·deprecated
  • SPEC_CONTEXT.md — 최종 상태 갱신
  • /speckit.analyze — 자동 실행

왜 마지막에 되돌아오는가?

SpecKit 문서 = 과거의 설계 의도  |  코드 = 현재의 구현 현실
반복 개선을 거치면 이 둘은 반드시 벌어짐 → 매 사이클마다 간극 제거

17단계는 전체 워크플로우의 마지막이자, 다음 사이클의 시작점을 정확하게 만드는 핵심 단계입니다. Bidirectional Feedback 원칙에 따라 코드 변경을 문서에 역반영합니다.

기능 규모별 접근 방식

모든 기능이 전체 플로우를 거칠 필요는 없습니다

S

소규모 — 단일 컴포넌트, 작은 로직 추가

start.bat (또는 npm run dev) + 대화형 → 완료 후 SPEC_CONTEXT.md 업데이트

M

중규모 — 여러 파일, UI 흐름 변경

start.bat (또는 npm run dev) + 대화형 → 10.5단계 (bkit 문서 동기화) → SPEC_CONTEXT.md 업데이트

L

대규모 — 새 API, 새 페이지, 새 DB 테이블

/pdca plan → /pdca design → /pdca do → bkit 풀 플로우

XL

초대규모 — 기존 구조 전면 개편

/speckit.specify → 풀 SpecKit → bkit 풀 플로우

기존 위에 추가 → /pdca plan  |  기존 구조를 뒤흔드는 변경 → /speckit.specify

규모 판단의 핵심 기준은 "기존 위에 추가하는가, 기존 구조를 뒤흔드는가"입니다. 소규모는 대화형으로 충분하고, 대규모부터 bkit 풀 플로우를 사용합니다.

speckit.analyze구현 전

pdca analyze구현 후
설계 vs 코드 갭 검증
