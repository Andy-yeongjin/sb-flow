sb-flow
워크플로우 가이드

bkit PDCA 기반 SDD(Spec-Driven Development)

헌법 + 디자인 시스템 강제 참조 → 계획 → 구현 → 검증 → 보고

sb-flow는 bkit PDCA를 중심으로 constitution.md 헌법과 디자인 시스템을 모든 단계에서 강제 참조하는 SDD 워크플로우입니다. 6단계로 단순화되어 빠르게 품질 높은 개발을 완주할 수 있습니다.

constitution.md

  • 세션 시작 시 즉시 읽기
  • 모든 /pdca 명령에 적용
  • SDD · DDD · 보안 · UX 원칙

Single Source of Truth

design.md + tokens.css

  • 하드코딩 절대 금지
  • 구현·검증·갭분석의 기준

bkit PDCA

sb-flow의 세 축은 헌법(constitution.md), 디자인 시스템(design.md+tokens.css), bkit PDCA입니다. 모든 개발 작업은 이 세 가지를 동시에 참조해야 합니다.

design.md — 디자인 명세

  • 색상 팔레트 정의
  • 타이포그래피 스케일
  • 컴포넌트 규칙
  • 레이아웃 그리드

design-tokens.css — CSS 변수

  • 모든 값을 --변수명 형태로
  • 구현 시 var(--토큰명)으로만 참조
  • 하드코딩(#hex, px값) 금지
  • 갭 분석 시 위반 감지

자동 교정: design-constitution.md(정부 UIUX 가이드라인) 기준으로 색상 대비·터치 타깃 등 위반 항목 즉시 교정
갭 분석 시: design.md + design-tokens.css 기준으로 하드코딩 위반 감지

/sb-design은 designs/stitch/ 폴더가 있으면 Stitch 모드, designs/*.pen이 있으면 Pencil 모드로 자동 분기합니다. 두 경우 모두 design-constitution.md 기준으로 자동 교정 후 design.md와 design-tokens.css를 생성합니다.

/sb-oneshot [기능 설명] — 위 6단계를 한 번에 자동 완주 (PRD·.pen 자동 감지)

6단계 워크플로우입니다. 1~3단계는 계획-설계-구현, 4~5단계는 검증-개선, 6단계는 브라우저 검증입니다. /sb-oneshot으로 전체를 자동화할 수 있습니다.

헌법 강제 참조 — 핵심 규칙

절대 금지: /pdca do {기능명} 한 줄만 실행 — 헌법·디자인 참조 누락

✅ 올바른 실행 형태 (3단계 코드 구현 예시):

/pdca do {기능명} constitution.md 파일을 읽고 모든 원칙을 반드시 준수해서 구현해줘. designs/design.md와 designs/design-tokens.css를 읽고, 모든 CSS 스타일링에서 var(--토큰명) 형태로 참조해줘. 하드코딩 금지. @vercel-react-best-practices 가이드라인을 읽고 이 기준에 맞춰 구현해줘.

❌ 하드코딩 금지

  • color: #2563EB → color: var(--color-primary-600)
  • height: 64px → height: var(--navbar-height)
  • padding: 16px → padding: var(--spacing-4)

✅ 토큰 참조 필수

  • design-tokens.css 정의 변수만 사용
  • 갭 분석에서 위반 자동 감지
  • Match Rate 90% 미달 시 /pdca iterate
가장 중요한 규칙입니다. /pdca 명령어 실행 시 반드시 constitution.md와 디자인 시스템을 함께 지시해야 합니다. /sb-oneshot이 이를 자동으로 처리합니다.

/sb-oneshot — 자동 파이프라인

Step 0 → 자산 감지: constitution.md (필수), design.md + tokens.css (필수), PRD (선택), .pen/Stitch (선택)
Step 1~3 → /pdca plan → design → do (헌법+디자인 참조 포함)
Step 4 → /pdca analyze — Match Rate 측정
Step 5 → Match Rate < 90% → /pdca iterate (최대 5회)
Step 6 → Playwright 브라우저 검증

PRD 자동 감지

prd/ 폴더의 .md 파일을 자동 탐색하여 개발 계획에 반영

.pen / Stitch 자동 감지

designs/*.pen 또는 designs/stitch/ 존재 시 UI 레이아웃 참조 (없어도 무방)

오류 처리

constitution.md 없으면 즉시 중단 및 안내 / design.md 없으면 /sb-design 먼저 실행 안내

/sb-oneshot은 6단계를 완전 자동화합니다. 핵심은 Step 0에서 자산을 감지하고, Step 1~3에서 헌법과 디자인 시스템을 강제 참조하는 것입니다. constitution.md가 없으면 실행을 중단합니다.

프로젝트 파일 구조

project-root/ ├── constitution.md # 개발 헌법 (필수) ├── designs/ │ ├── design.md # 디자인 명세 (필수) │ ├── design-tokens.css # CSS 변수 (필수) │ ├── design-constitution.md # 정부 UIUX 기준 │ ├── *.pen # Pencil 파일 (선택) │ └── stitch/ # Stitch ZIP 압축 해제 폴더 (선택) ├── prd/ # PRD 문서 (선택) │ └── feature.md ├── docs/ # bkit 생성 문서 │ ├── 01-plan/ │ └── 02-design/ └── src/ # 소스 코드

필수 파일

  • constitution.md — 루트에 위치, 세션마다 읽기
  • designs/design.md — 디자인 명세 본체
  • designs/design-tokens.css — CSS 변수 집합

선택 파일 (자동 감지)

  • designs/*.pen — UI 레이아웃 원본 (Pencil)
  • designs/stitch/ — UI 레이아웃 원본 (Stitch)
  • prd/*.md — 기능 요구사항
  • design-constitution.md — sb-design에서만 사용
파일 구조입니다. constitution.md는 프로젝트 루트에, .pen 파일은 반드시 designs/ 폴더에 위치해야 합니다. design-constitution.md는 sb-design 명령어에서만 사용됩니다.

자주 묻는 질문 (FAQ)

Q. constitution.md는 어디에?

프로젝트 루트에 위치합니다. sb-flow setup.bat 실행 시 자동으로 복사됩니다. 세션 시작마다 CLAUDE.md 지시에 따라 AI가 자동으로 읽습니다.

Q. 디자인 파일이 없어도 되나요?

네. 없으면 /sb-design이 design-constitution.md 기반으로 기본 디자인 시스템을 생성합니다. Pencil.dev라면 designs/ 폴더에 .pen 파일을, Stitch라면 designs/stitch/ 폴더에 ZIP 압축 해제 후 넣으면 자동 감지됩니다.

Q. Match Rate가 90% 미만이면?

/pdca iterate를 실행합니다 (최대 5회). /sb-oneshot은 이를 자동으로 처리합니다. 90% 이상이 되면 /simplify → /pdca report 순서로 마무리합니다.

Q. design-constitution.md는 언제 쓰나요?

/sb-design 실행 시에만 사용합니다. 정부 UIUX 가이드라인(색상 대비, 터치 타깃 등) 준수 여부를 토큰 생성 시 검증하는 용도입니다. 구현 단계에서는 사용하지 않습니다.

💡 모르면 /sb-guide — 현재 단계를 자동 감지하고 다음 실행 명령어를 안내합니다

가장 많이 물어보는 4가지 질문에 대한 답변입니다. 핵심은 constitution.md는 루트에, .pen은 designs/에, design-constitution.md는 sb-design에서만 사용한다는 것입니다.
