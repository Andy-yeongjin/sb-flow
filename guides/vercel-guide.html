1

GitHub 저장소 만들기 사람

Vercel은 GitHub 저장소를 보고 자동 배포합니다. 빈 저장소만 만들어 두세요.

GitHub 미가입자는 github.com/signup 에서 먼저 가입

로그인 후 우측 상단 + → New repository 클릭

아래 항목 입력 → Create repository

  • Repository name: 프로젝트 이름 (예: my-app)
  • Public / Private: 어느 쪽이든 OK (Vercel 무료 플랜은 둘 다 지원)
  • Initialize with README/.gitignore/license: 전부 체크 해제 — 빈 저장소로 만들어야 충돌 없습니다

생성 후 표시되는 화면에서 파란색 영역의 저장소 URL을 복사해 둡니다. 형태:

예시
https://github.com/yourname/my-app.git
2

Claude에게 푸시 시키기 프롬프트 복사

아래 프롬프트의 {{REPO_URL}}만 1단계에서 복사한 URL로 교체.

Claude Code 대화창에 붙여넣기 → REPO_URL 교체
constitution.md를 읽고 모든 원칙 준수. 특히 보안: .env가 .gitignore에 포함되어 있는지 반드시 먼저 확인하고, 자격증명이 한 번도 커밋된 적이 없는지 git log로 검증해줘. 이 프로젝트를 GitHub에 푸시할 거야. 아래 절차를 순서대로 수행해: 1. .gitignore 점검: - .env, .env*.local, node_modules, .next, .vercel 가 포함되어 있는지 확인 - 누락된 항목은 추가 2. .env.example 파일 생성: - 현재 .env에 있는 모든 키를 가져오되 값은 모두 빈 문자열로 - 각 키 위에 한 줄 주석으로 어디서 받는 값인지 표기 - (예: # NeonDB → Connection pooling string) 3. Vercel 배포 준비: - package.json scripts.build를 "prisma generate && next build" 로 수정 (Prisma 사용 중일 때만. Drizzle 등 다른 ORM이면 그에 맞게) - postinstall 스크립트가 필요하면 "prisma generate" 추가 - next.config.js / next.config.ts 검토 — Vercel에서 깨질 만한 설정 없는지 확인 4. README.md 갱신 (없으면 생성): - 프로젝트 한 줄 설명 - 로컬 실행 방법 (npm install → .env.example을 .env로 복사 후 값 채우기 → npm run dev) - 사용 중인 환경변수 목록과 출처 5. git 초기화 및 푸시: - 현재 디렉터리가 git 저장소가 아니면 git init - 기본 브랜치를 main으로 설정 (git branch -M main) - origin 추가: git remote add origin {{REPO_URL}} - 첫 커밋 생성 후 git push -u origin main 6. 푸시 후 git status로 작업 트리 깨끗한지 확인하고, .env 파일이 절대 트래킹되지 않은 상태인지 git ls-files로 검증해줘. 작업 종료 후 어떤 파일이 추가/변경됐는지, .env가 안전한지 한 줄로 보고해줘.
🚨
푸시 후 GitHub 저장소 페이지를 직접 새로고침해서 .env 파일이 보이지 않는지 눈으로 확인하세요. 보이면 즉시 NeonDB 비밀번호 재발급 + git history 정리.
3

Vercel 가입 + 프로젝트 Import 사람

GitHub 계정으로 한 번에 가입 + 저장소 권한까지 연결됩니다.

vercel.com/login → Continue with GitHub → 권한 승인

대시보드 우측 상단 Add New... → Project

Import Git Repository 목록에서 2단계에서 푸시한 저장소 옆 Import 클릭

방금 만든 repo가 목록에 안 보일 때 — Vercel GitHub App이 새 repo에 접근 권한이 없어서 그래요. 아래 순서로 권한을 추가합니다.

  1. Import 화면 아래쪽 "Adjust GitHub App Permissions" 또는 "Configure GitHub App" 링크 클릭 → GitHub로 이동
  2. 본인 계정 옆 Configure 클릭
  3. Repository access 섹션에서 두 가지 중 선택:
    • All repositories 선택 (간단함, 향후 만들 repo도 자동 포함) — 권장
    • 또는 Only select repositories → Select repositories 드롭다운에서 방금 만든 repo 추가
  4. 아래 Save 클릭
  5. Vercel Import 화면으로 돌아와 새로고침 → 이제 repo가 목록에 보임
⚠️
GitHub 조직(Organization)의 repo면 그 조직 옆 Configure를 눌러야 합니다. 본인 계정만 설정해서는 조직 repo가 안 보여요.

Configure Project 화면에서 Framework Preset이 Next.js 로 자동 감지됐는지 확인. 빌드 명령은 건드리지 마세요 (2단계에서 이미 package.json에 설정).

확인했으면 화면 하단 Deploy 버튼 클릭 (Environment Variables 섹션은 비워둔 채로).

4

배포 결과 확인 자동 배포

1~3분 정도 기다리면 빌드·배포가 완료되고 도메인이 발급됩니다.

빌드 로그가 실시간으로 흐릅니다. 1~3분 후 🎉 Congratulations 화면 + 배포 URL 표시

✅
표시된 URL(https://my-app-xxxx.vercel.app)이 이 프로젝트의 배포 도메인입니다. 클릭해서 사이트가 뜨는지 확인하세요.
⚠️
이 첫 배포는 사이트 접속 시 DB 연결 실패가 정상입니다. 환경변수가 아직 없어서 그래요. 5단계에서 등록 후 Redeploy하면 정상 동작합니다. (빌드 자체가 실패한 경우만 FAQ 참고)
5

환경변수 등록 + Redeploy 사람

배포된 프로젝트 → Settings → Environment Variables로 이동.

아래 표대로 값을 입력 — Key/Value 한 쌍씩 Add:

KEY
출처 (어디서 복사?)
환경
DATABASE_URL
NeonDB → Connection Details → Pooled URL
All
DIRECT_URL
NeonDB → Connection Details → Direct URL (Pooled OFF)
All
🔁
.env 에 다른 키가 더 있다면 같은 방식으로 모두 등록하세요. (예: 외부 API 키, OAuth client ID/secret 등)

모든 키 등록 후 Deployments 탭 → 최신 배포 우측 ⋯ → Redeploy

✅
Redeploy가 끝나면 발급된 URL을 다시 열어 정상 동작 확인. DB 연결·인증 모두 살아 있어야 합니다.
6

도메인 확인 + 이후 자동 배포 완료

지금부터는 git push만 하면 자동 재배포됩니다.

커스텀 도메인이 있다면 Settings → Domains에서 추가 + DNS 안내대로 A/CNAME 레코드 등록

이후 작업 흐름은 Claude로 코드 수정 → git push 이게 전부. Vercel이 GitHub webhook을 받아 자동 빌드·배포합니다.

코드 수정 후 Claude에게 — 한 줄이면 충분
방금 작업 내용을 의미 있는 단위로 커밋하고 origin main에 푸시해줘. 커밋 메시지는 변경사항을 한 줄로 요약하고, push 후 git log -1로 결과 확인해서 요약해줘.

Pull Request 단위 미리보기가 자동으로 켜져 있습니다. main 외 브랜치를 푸시하면 별도 Preview URL이 발급되고, PR에 봇이 코멘트로 링크를 남깁니다.

자주 묻는 질문 / 트러블슈팅

빌드 로그에 "PrismaClientInitializationError" ▼
① Vercel Environment Variables에 DATABASE_URL / DIRECT_URL 둘 다 등록됐는지 확인
② package.json scripts.build가 "prisma generate && next build" 인지 확인
③ NeonDB 무료 플랜 슬립 상태일 수 있음 — 1분 후 Redeploy
"Module not found: @prisma/client" ▼
postinstall 스크립트에 "prisma generate"를 추가하거나, build 스크립트 앞에 붙이세요. 2단계 프롬프트에 이미 포함되어 있지만 직접 만든 프로젝트라면 누락됐을 수 있습니다.
DB 마이그레이션은 어떻게 배포해요? ▼
로컬에서 npx prisma migrate dev로 마이그레이션 파일을 생성·커밋·푸시한 뒤, build 스크립트를 "prisma migrate deploy && prisma generate && next build" 로 바꾸면 Vercel 빌드 시 자동 적용됩니다. 다만 빌드 실패 시 롤백이 안 되므로, 운영 단계에서는 별도 마이그레이션 단계를 분리하는 것을 권장합니다.
환경변수를 나중에 바꾸면? ▼
Settings → Environment Variables에서 수정한 뒤, 반드시 Redeploy해야 적용됩니다 (이미 빌드된 함수는 옛 값을 들고 있음). Deployments → 최신 배포 우측 ⋯ → Redeploy.
"Function timed out" 또는 응답이 너무 느려요 ▼
① NeonDB 리전과 Vercel 함수 리전이 가까운지 확인 (NeonDB ap-southeast-1 (Singapore) ↔ Vercel sin1 권장)
② Vercel 무료 플랜 함수 타임아웃은 10초. 무거운 작업은 background job으로 분리하거나 Hobby → Pro 업그레이드 고려
③ Pooled URL을 쓰고 있는지 확인 (Direct URL을 런타임에서 쓰면 풀 누수 발생)
실수로 .env를 푸시했어요 ▼
1. 즉시 자격증명 무효화: NeonDB Settings → Reset password (다른 키도 모두 회전)
2. git history 정리: git filter-repo --path .env --invert-paths 후 force push. 또는 새 저장소로 옮기기
3. .gitignore 보강 후 새 자격증명으로 .env 재작성
파일을 단순히 git에서 지우는 것만으로는 history에 남아 GitHub 검색·캐시·포크에 그대로 노출됩니다.
Vercel 무료 플랜 한계 ▼
개인 사용·취미 프로젝트만 허용 (상업적 사용 금지), 100 GB/월 대역폭, 함수 실행 10초, 6,000 빌드 분/월. 사이드 프로젝트·MVP에는 충분합니다. 회사 제품이라면 Pro 플랜($20/월) 필요.
